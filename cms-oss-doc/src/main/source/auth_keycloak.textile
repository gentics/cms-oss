h2. Keycloak

"Keycloak":http://www.keycloak.org/ is an identity and access management system
providing a trusted way for an application to authenticate a user via different
authentications protocols.

endprologue.

NOTE: Gentics CMS supports Keycloak versions 19.x to 24.x.

h3. Server side configuration

Activate the Keycloak feature and add configuration for user creation callbacks and initial groups:

<shell filename="keycloak.yml">
feature:
  keycloak: true
keycloak:
  user_created_callback: your.companyname.custom.SsoUserCreated
  sync_groups: true
  init_groups:
    roles.realm.realm-role-1: RealmRoleGroup1
    roles.realm.realm-role-2:
      - RealmRoleGroup2a
      - RealmRoleGroup2b
    roles.realm.realm-role-3:
      - RealmRoleGroup1
      - RealmRoleGroupX
      - group: RealmRoleGroup3
        nodes: Node 3
    roles.resource.cms.cms-role:
      - group: CmsRoleGroup
        nodes:
          - Node 1
          - 4488.c0767dce-f8fd-11ef-8735-769f09e19595
          - NonExistantNode
</shell>

h4. Configuration options

The following configuration entries define how SSO login via Keycloak is handled.

1. @user_created_callback@: An implementation of the @SsoUserCreatedCallback@ interface which must implement only the method @void accept(User user, Map<String, Object> attributes)@, where @user@ is the user that was created automatically via an SSO login and @attributes@ is a collection of user information provided by the concrete SSO implementation (e.g. the contents of the users access token from Keycloak).

2. @sync_groups@: Whether an already existing users groups should be checked everytime an SSO login is performed. If the attributes of the user changed (e.g. the assigned roles in Keycloak), the groups of the user will be changed to match the new attributes.

3. @init_groups@: Mappings from user attributes to lists of CMS user groups. In the example above, this means that if @roles.realm.realm-role-1@ can be resolved in the user attributes, the created user will be assigned to the group @RealmRoleGroup1@. The groups may be a single group name or a list of groups. In a list of groups each entry may be a single string which must be the name of the group, or a mapping with the keys @group@ and @nodes@ which hava the group name and a list of nodes to which the group assignment is restricted respectively. Nodes may be specified by their name or their global ID. Non-existing groups and nodes will produce a warning in the logs, but will otherwiese be ignored.

Place the `keycloak.json` generated for the CMS in the UI configuration directory (default: `ui-conf`).

h3. Configuration for the Editor/Administration User Interface

If you are using the new UI, you will also have to make sure the REST-API
is configured to use the "Keycloak servlet filter":rest_auth.html#sso-with-keycloak.

Except for the @groupMapper@ and @userCreatedCallback@ entries, the configuration
entries correspond exactly to the respective fields in the @keycloak.json@ you can
generate in the Keycloak admin front-end, and you should put all the entries here even
when they are not listed in the example above.

h4. Setup Content Security Policy

The Administrator User Interface requires some additional settings in Keycloak, to work as expected.
Update the "Content Security Policy" of your Keycloak realm:

# Select your Realm in the top left corner
# Select the tab "Realm Settings" in the "Configure" category on the left sidebar
# Select the tab "Security Defenses" on the top bar

Adopt the "Content-Security-Policy" to allow the embedding from your CMS installation host:

<shell>
frame-src 'self'; frame-ancestors 'self'; object-src 'none';
frame-src 'self'; frame-ancestors 'self' https://yourcmshost.tld; object-src 'none';
</shell>

h4. Using an Identity Provider

When using an "identity provider":https://www.keycloak.org/docs/latest/server_admin/#_identity_broker
the following two settings in the advanced OpenID Connect settings are important to prevent unwanted
redirect behavior:

# "Prompt": Set to "Select account"
# "Accepts prompt=none forward from client": Set to "Off"

h4. Circumvent Single Sign-on

It can usefull to not use the single sign process although it is enabled.
The single sign on process will be skipped and users will be forwarded to the login form when the url parameter @skip-sso@ is present.
Example: @https://yourcmshost.tld/editor/?skip-sso@

h4. Single Sign Out

By activating the feature

<javascript filename="conf/features.yml">
feature:
	keycloak_signout: true
</javascript>

users will automatically be logged out from Keycloak, if they log out from the CMS using the Editor User Interface or Administration User Interface.
